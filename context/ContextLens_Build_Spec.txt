Yes — below is a build-ready product spec for **ContextLens**, including features, system design, data model, APIs, UX, delivery plan, and acceptance criteria. The core architecture is realistic because VS Code extensions can provide both sidebar tree views and fully custom webviews, while Firebase AI Logic supports multi-turn Gemini interactions, structured outputs, and production protections like per-user rate limits and App Check.

## Product scope

ContextLens should be positioned as an **AI workflow memory layer** for coding, not as another chatbot or IDE replacement. The product should make every meaningful AI-assisted coding action traceable, reviewable, and reusable across a branch, a task, and eventually a team.

### Product definition

| Item | Spec |
|---|---|
| Product name | ContextLens |
| One-line pitch | “A Git-aware AI context layer that turns coding sessions into searchable episodes.” |
| Primary users | AI-heavy developers, indie hackers, student teams, tech leads reviewing AI-generated changes |
| Core job to be done | “Help me remember what the AI changed, why it changed it, and how to explain that to myself or my reviewer.” |
| MVP surfaces | VS Code extension + web dashboard |
| Core differentiator | Persistent episode timeline linked to git state, prompts, responses, and summaries |
| Non-goals for MVP | Full IDE replacement, team collaboration suite, GitHub App, JetBrains plugin, mobile app |

### Product goals

- Reduce context loss during AI-assisted coding.
- Make branch evolution understandable in under 60 seconds.
- Help developers generate review-ready explanations from actual coding history.
- Create a demo that instantly looks technical, useful, and premium.

### MVP success criteria

- A developer can create an episode, send an AI request, and see it logged with branch + file + diff.
- A developer can open the dashboard and inspect the timeline of calls in that episode.
- A developer can click “Explain Diff” and get a readable summary with risks and next checks.
- A developer can click “Summarize Branch” and get a PR-style overview.

## Features and UX

A sidebar state panel is straightforward in VS Code because the platform explicitly supports custom tree views in the sidebar, and richer interactive panes can be built with webviews. That means the best MVP UI is a lightweight tree-based “state header” plus one richer webview for chat and detailed interaction.

### Feature set

| Feature | User value | MVP spec | Stretch spec |
|---|---|---|---|
| Episode management | Organizes work into task-sized chunks | Create episode, rename episode, close episode, switch active episode | Auto-suggest episode names from branch + prompt |
| AI chat logging | Preserves AI decisions | Built-in Gemini chat panel; every request logs metadata | Log external AI calls from pasted prompt/response |
| Git context capture | Anchors AI output to real code state | Capture branch, active file, diff snapshot at request time | Capture changed file list and commit hashes |
| Intent tagging | Makes timeline readable | Optional short intent field before/after each call | Auto-classify intent from prompt |
| State header | Gives live situational awareness | Current episode, branch, call count, AI-owned files | “Suggested next action” card |
| Episode timeline | Makes history searchable | Dashboard list of episodes with expandable calls | Timeline heatmap, branch analytics |
| Explain Diff | Helps self-review and code review | Gemini summary + risks + checks for selected diff | Inline test suggestions |
| Branch summary | Converts work into presentation-ready narrative | PR summary, key changes, risk notes | Markdown export directly to GitHub |
| Search and filter | Makes history useful later | Search by branch, file, intent, date | Search by semantic similarity |
| Auth and projects | Separates repos/workspaces | Google sign-in, project-per-repo | Org/workspace sharing |

### User flows

#### 1) New episode flow
1. Developer clicks `New Episode`.
2. Enters label, for example `Refactor auth middleware`.
3. Extension stores active episode locally and starts routing all new calls into it.

#### 2) AI request flow
1. Developer opens chat in the sidebar.
2. Types prompt and optional intent tag.
3. Extension captures current branch, active file, and diff snapshot.
4. Backend logs the call, sends it to Gemini, stores the response, and returns output.
5. State header increments call count immediately.

#### 3) Explain diff flow
1. Developer opens an episode or current state.
2. Clicks `Explain Diff`.
3. Backend assembles diff + intent + file context.
4. Gemini returns a review-style explanation.
5. Result is stored and shown in both the extension and dashboard.

#### 4) Branch summary flow
1. User opens a branch in the dashboard.
2. Clicks `Summarize Branch`.
3. Backend aggregates episode summaries and major calls.
4. Gemini outputs PR-ready text with risks and follow-up checks.

## System design

Firebase AI Logic supports multimodal prompting, multi-turn chat, structured output, and configurable per-user rate limits, so it fits both the interactive IDE chat path and the summary-generation endpoints. Firebase AI Logic also keeps the Gemini API key on Google-managed servers rather than exposing it to the client, and App Check adds an extra abuse-protection layer for production hardening.

### Architecture

| Layer | Tech | Responsibility |
|---|---|---|
| IDE client | VS Code extension, TypeScript | Capture context, render sidebar/webview, send requests |
| Web app | Next.js + Firebase Hosting | Timeline, search, episode detail, branch summary |
| Auth | Firebase Auth | Sign-in with Google |
| Database | Firestore | Projects, episodes, calls, summaries, settings |
| Backend | Firebase Functions or Cloud Run | Logging, summarization, aggregation, model proxy |
| AI layer | Gemini via Firebase AI Logic | Chat, explain diff, branch summary, structured outputs |
| Optional protection | Firebase App Check | Anti-abuse, production hardening |

### Recommended repo structure

```text
contextlens/
  apps/
    extension/           # VS Code extension
    dashboard/           # Next.js app
  services/
    api/                 # Firebase Functions or Cloud Run handlers
  packages/
    shared-types/        # TS types for Episode, Call, Summary
    prompt-templates/    # Gemini prompts
    ui/                  # Shared UI tokens if needed
  docs/
    architecture/
    demo/
    product/
```

### Extension module breakdown

| Module | Responsibility |
|---|---|
| `extension.ts` | Activation, command registration, lifecycle |
| `auth.ts` | Sign-in, token retrieval, session persistence |
| `episodeStore.ts` | Active episode state, local caching |
| `gitContext.ts` | Branch detection, diff capture, file list |
| `chatViewProvider.ts` | Webview chat UI |
| `stateTreeProvider.ts` | Sidebar tree view |
| `apiClient.ts` | Calls backend endpoints |
| `telemetry.ts` | Local timing/events |
| `redaction.ts` | Secret detection before upload |

### Backend services

| Endpoint | Method | Purpose | Input | Output |
|---|---|---|---|---|
| `/projects/create` | POST | Create project | repo name, repo URL | project id |
| `/episodes/create` | POST | Start episode | project id, label, branch | episode id |
| `/calls/log` | POST | Log AI call | project id, episode id, prompt, context | stored call + model output |
| `/episodes/explain` | POST | Explain accumulated diff | episode id | summary, risks, checks |
| `/branches/summarize` | POST | Summarize branch | project id, branch | PR summary |
| `/search` | POST | Search timeline | query, filters | episodes/calls |
| `/episodes/close` | POST | Close episode | episode id | status updated |

### AI task design

| Task | Model mode | Output shape |
|---|---|---|
| IDE chat | Fast interactive Gemini model | plain text + citations/notes if needed |
| Explain diff | Higher-quality reasoning model | JSON: `summary`, `risks[]`, `checks[]` |
| Episode summary | Reasoning model | JSON: `what_changed`, `why`, `files[]` |
| Branch summary | Reasoning model | JSON: `pr_summary`, `key_changes[]`, `review_risks[]` |
| Intent labeling | Fast model or rule-based | short label string |

### Key engineering decisions

- Use **structured JSON outputs** from Gemini for all summary endpoints.
- Treat the extension as a thin client; keep AI prompts and aggregation logic on the server.
- Cache summaries by `episodeId + diffHash` so repeated clicks are instant.
- Upload only bounded context per request; do not send the whole repository.
- Build for one IDE first: VS Code only.

## Data model and technical specs

Firestore organizes data hierarchically, and security rules apply only to the exact matched path, not automatically to subcollections, so the schema and rules must explicitly cover nested episode and call collections. That makes a user-scoped hierarchy like `users/{uid}/projects/{projectId}/episodes/{episodeId}/calls/{callId}` the safest default for the MVP.

### Firestore schema

```text
users/{uid}
  displayName
  email
  photoURL
  createdAt

users/{uid}/projects/{projectId}
  name
  repoUrl
  localWorkspaceName
  defaultBranch
  createdAt
  updatedAt
  settings:
    preferredModel
    redactionEnabled
    autoSummariesEnabled

users/{uid}/projects/{projectId}/episodes/{episodeId}
  label
  branchName
  status                # active | closed
  startedAt
  endedAt
  callCount
  changedFiles[]
  latestDiffHash
  manualNotes
  episodeSummary
  explainDiffSummary
  explainDiffRisks[]
  explainDiffChecks[]

users/{uid}/projects/{projectId}/episodes/{episodeId}/calls/{callId}
  createdAt
  source                # extension_chat | manual_log
  intentTag
  promptText
  modelName
  modelResponse
  branchName
  activeFilePath
  relatedFiles[]
  diffSnapshot
  diffHash
  todoMatches[]
  latencyMs
  tokenUsage:
    input
    output
  status                # success | failed | redacted
```

### Local extension state

| Key | Type | Purpose |
|---|---|---|
| `activeProjectId` | string | Current repo mapping |
| `activeEpisodeId` | string | Current task context |
| `authSession` | secure token/session data | Auth reuse |
| `draftIntent` | string | Optional convenience field |
| `lastDiffHash` | string | De-duplicate repeated logs |
| `queuedRequests` | array | Retry on network failure |

### Suggested security rules shape

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /users/{uid} {
      allow read, write: if request.auth != null && request.auth.uid == uid;

      match /projects/{projectId} {
        allow read, write: if request.auth != null && request.auth.uid == uid;

        match /episodes/{episodeId} {
          allow read, write: if request.auth != null && request.auth.uid == uid;

          match /calls/{callId} {
            allow read, write: if request.auth != null && request.auth.uid == uid;
          }
        }
      }
    }
  }
}
```

### Prompt templates

#### Explain Diff prompt
```text
You are a senior software engineer reviewing an AI-assisted code change.

Inputs:
- Episode label: {label}
- Branch: {branchName}
- Intents: {intentTags}
- Changed files: {files}
- Unified diff: {diff}

Return strict JSON:
{
  "summary": "2-4 sentence explanation",
  "risks": ["risk 1", "risk 2"],
  "checks": ["check 1", "check 2", "check 3"]
}

Focus on:
1. What changed
2. Why it likely changed
3. What might break
4. What should be tested
```

#### Branch Summary prompt
```text
You are preparing a pull request summary from an ordered sequence of AI-assisted coding episodes.

Inputs:
- Branch: {branchName}
- Episode summaries: {episodes}
- Changed files: {changedFiles}

Return strict JSON:
{
  "pr_summary": "3-5 sentences",
  "key_changes": ["...", "..."],
  "review_risks": ["...", "..."]
}
```

### Error states to handle

| Error | Extension behavior | Dashboard behavior |
|---|---|---|
| Not a git repo | Disable diff capture, allow manual logging | Show “No git context available” |
| No active episode | Prompt to create episode before chat | Highlight missing context |
| AI request timeout | Retry button, keep request draft | Show pending state |
| Firestore write fail | Queue locally and retry | Mark event as unsynced |
| Model quota exceeded | Fallback message, suggest retry later | Disable summary actions temporarily |
| Secret detected in diff | Ask user to redact or skip upload | Mark call as redacted |

## Build plan and acceptance criteria

Firestore best-practice guidance emphasizes rules and data-model planning up front, so the fastest way to ship this is to lock the schema, security rules, and endpoint contracts before polishing UI. You should also keep the model layer configurable because Firebase AI Logic’s supported-model lineup changes over time, including published shutdown notices for older Gemini variants.

### Delivery phases

| Phase | Goal | Output |
|---|---|---|
| Phase 1 | Foundation | Firebase project, auth, Firestore schema, extension skeleton |
| Phase 2 | Core logging | Create episode, log call, capture branch/file/diff |
| Phase 3 | AI value loop | Explain Diff and Branch Summary working end-to-end |
| Phase 4 | Dashboard | Timeline, filters, episode detail, branch page |
| Phase 5 | Polish | dark mode, loading states, redaction, demo data |
| Phase 6 | Submission prep | README, architecture diagram, demo video, screenshots |
